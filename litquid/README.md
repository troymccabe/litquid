# LitQuid Toolchain & Samples

Bridge tooling and sample applications for LitQuid.

## Toolchain

### Rust Preprocessor (`litquid`)

Converts `.liquid` templates to Lit template modules and optional server-side renderers.

```bash
cd toolchain
cargo build --release

# JS output only (default)
cargo run --bin litquid -- \
  --input ./path/to/templates \
  --output ./path/to/output

# JS + C# server-side renderers
cargo run --bin litquid -- \
  --input ./path/to/templates \
  --output ./path/to/output \
  --emit csharp \
  --namespace MyApp.Generated

# Watch mode — regenerates all outputs on .liquid changes
cargo run --bin litquid-watch -- \
  --input ./path/to/templates \
  --output ./path/to/output \
  --emit csharp \
  --namespace MyApp.Generated
```

Options:
- `--emit <targets>` - Server-side render targets. Supported: `csharp`
- `--namespace <ns>` - Namespace for generated server-side code (default: `LitQuid.Generated`)
- `--lit-import <path>` - Customize lit import path (default: `"lit"`)
- `--port <port>` - Live reload server port, watch mode only (default: `35729`)

Add the live reload script to your HTML:

```html
<script src="http://localhost:35729/livereload.js"></script>
```

### What the Preprocessor Emits

For each `.liquid` template the preprocessor writes up to three files:

| File | Always? | Purpose |
|---|---|---|
| `my-component.template.js` | Yes | `getTemplate(ctx)` function for client-side Lit rendering |
| `my-component.template.json` | Yes | Digest manifest consumed by the Liquid SSR path |
| `my-component.template.cs` | With `--emit csharp` | Typed `Render(...)` method for the Generated SSR path |

### How the Preprocessor Works

1. Parses `.liquid` files
2. Finds `{{ ... | csr }}` expressions and extracts or auto-translates the client-side JS expression
3. Computes the Lit template digest (DJB2-inspired over the static HTML segments)
4. Writes `.template.js` (Lit `html` tagged template), `.template.json` (digest manifest)
5. If `--emit csharp`: writes a C# class with the digest as a compile-time constant and a `Render(...)` method using `string.Concat` — no runtime template engine required

**Input:**
```liquid
<template shadowrootmode="open">
  <p class="name">{{ firstName | capitalize | csr }}</p>
  <p class="meta">{{ role | csr }} · {{ age | csr }} yrs</p>
</template>
```

**`.template.js` output:**
```javascript
import { html } from 'lit';

export const templateDigest = 'mKjZkQEA/6Y=';

export function getTemplate(ctx) {
  return html`
    <p class="name">${ctx.firstName.charAt(0).toUpperCase() + ctx.firstName.slice(1).toLowerCase()}</p>
    <p class="meta">${ctx.role} · ${ctx.age} yrs</p>`;
}
```

**`.template.cs` output (with `--emit csharp`):**
```csharp
// Generated by litquid — do not edit.
namespace MyApp.Generated;

public static class MyComponentTemplate
{
    public const string Digest = "mKjZkQEA/6Y=";

    public static string Render(string firstName, string role, string age)
    {
        return string.Concat(
            "<template shadowrootmode=\"open\">\n<!--lit-part ", Digest, "-->\n",
            "  <p class=\"name\"><!--lit-part-->", Capitalize(firstName), "<!--/lit-part--></p>\n",
            "  <p class=\"meta\"><!--lit-part-->", role, "<!--/lit-part--> · ",
            "<!--lit-part-->", age, "<!--/lit-part--> yrs</p>\n",
            "<!--/lit-part-->\n</template>"
        );
    }

    private static string Capitalize(string s) =>
        string.IsNullOrEmpty(s) ? s : char.ToUpper(s[0]) + s.Substring(1).ToLower();
}
```

## SSR Paths

LitQuid supports two server-side rendering strategies. Both produce identical HTML with the same `<!--lit-part DIGEST-->` markers, so client hydration is identical regardless of which path was used.

### Liquid Path (runtime)

Uses the Liquid engine + `csr` filter at request time. The filter wraps each dynamic value in `<!--lit-part-->value<!--/lit-part-->` markers; `TemplateService` reads the digest from the `.template.json` manifest and injects the outer `<!--lit-part DIGEST-->` wrapper.

```
.liquid file + data
  → Liquid engine (runtime)
      → csr filter: wraps each binding in <!--lit-part--> markers
  → TemplateService.RenderSsrComponentAsync()
      → reads digest from .template.json
      → injects <!--lit-part DIGEST-->...<!--/lit-part--> around shadow root content
  → complete <template shadowrootmode="open"> string
```

**Trade-offs:** Full Liquid template language (loops, conditionals, includes, custom filters). Requires the Liquid engine and `csr` filter to be installed; manifest read adds one async I/O per unique template (cached after first use).

### Generated Path (build-time)

The preprocessor emits a typed render function with the digest baked in as a compile-time constant. No template engine, no manifest read, no async I/O — just `string.Concat`.

```
.liquid file
  → Rust preprocessor (build time) → MyComponentTemplate.Render(...)

At runtime:
  MyComponentTemplate.Render(firstName, role, age)
  → returns complete <template shadowrootmode="open"> string
```

**Trade-offs:** Fastest possible SSR; zero runtime dependencies beyond the generated file. Limited to data binding — no runtime conditionals or loops inside the template. Digest is a compile-time constant; regenerate by re-running the preprocessor when the template changes.

## Samples

### Shared Components

The `components/` directory contains shared component definitions used by all samples:

- `templates/` - `.liquid` template files (SSR source of truth)
- `scripts/` - TypeScript Lit component definitions

### .NET Sample

A complete ASP.NET Core sample demonstrating all three rendering strategies on a single page:

- **SSR · Generated** — `StatCardTemplate.Render()` and `MyComponentTemplate.Render()` with digest as compile-time constant
- **SSR · Liquid** — `TemplateService.RenderSsrComponentAsync()` with DotLiquid + `csr` filter
- **CSR** — attribute-only elements; empty shadow DOM in HTTP response

```bash
cd samples/dotnet
dotnet run
```

The `Generated/` directory is populated at build time by the npm build script, which invokes the Rust preprocessor with `--emit csharp`. MSBuild auto-discovers all `*.cs` files, so no `.csproj` changes are needed when templates are added or removed.
