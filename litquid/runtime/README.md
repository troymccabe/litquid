# @litquid/runtime

LitQuid runtime — SSR hydration support for Lit components.

## Installation

```bash
npm install @litquid/runtime
```

## Usage

```typescript
import { LitQuidElement, css, type TemplateResult } from '@litquid/runtime';
import { getTemplate } from './my-component.template.js';

class MyComponent extends LitQuidElement {
    static properties = {
        name: { type: String },
        age:  { type: Number },
    };

    static styles = css`
        :host { display: block; }
    `;

    name = '';
    age  = 0;

    protected renderTemplate(): TemplateResult {
        return getTemplate(this);
    }
}

customElements.define('my-component', MyComponent);
```

Always import `lit` primitives from `@litquid/runtime`, never from `'lit'` directly. The runtime ensures `@lit-labs/ssr-client/lit-element-hydrate-support.js` is patched before `LitElement` initialises, regardless of bundler order.

## How It Works

`LitQuidElement` extends `LitElement` and imports `@lit-labs/ssr-client/lit-element-hydrate-support.js` as a side-effect. That module patches `LitElement` to call `hydrate()` instead of `render()` on the first update when the shadow root already contains `<!--lit-part DIGEST-->` markers.

### SSR paths

LitQuid supports two server-side rendering strategies. The client side is identical for both.

**Liquid path** — the Liquid engine renders the `.liquid` template at request time; the `csr` filter wraps each binding in `<!--lit-part-->` markers; `TemplateService` injects the outer `<!--lit-part DIGEST-->` wrapper using the digest from the `.template.json` manifest:

```html
<!-- Server response -->
<my-component ssr name="Alice" age="28">
    <template shadowrootmode="open">
        <!--lit-part mKjZkQEA/6Y=-->
          <p><!--lit-part-->Alice<!--/lit-part-->, <!--lit-part-->28<!--/lit-part--> yrs</p>
        <!--/lit-part-->
    </template>
</my-component>
```

**Generated path** — the Rust preprocessor emits a typed render function at build time with the digest baked in as a compile-time constant. No template engine, no manifest read:

```csharp
// Generated by litquid — do not edit.
public static class MyComponentTemplate
{
    public const string Digest = "mKjZkQEA/6Y=";

    public static string Render(string name, string age) => string.Concat(
        "<template shadowrootmode=\"open\">\n<!--lit-part ", Digest, "-->\n",
        "  <p><!--lit-part-->", name, "<!--/lit-part-->, <!--lit-part-->", age, "<!--/lit-part--> yrs</p>\n",
        "<!--/lit-part-->\n</template>"
    );
}
```

Both produce the same HTML. `@lit-labs/ssr-client` hydrates either variant identically — the client does not know or care which path was used.

### Hydration flow

1. Browser parses the HTML, adopts `<template shadowrootmode="open">` into the shadow root
2. `LitElement.performUpdate()` detects `<!--lit-part DIGEST-->` in the shadow root and calls `hydrate()` instead of `render()`
3. `hydrate()` traverses the comment nodes, connects each `<!--lit-part-->` region to the corresponding Lit binding
4. Lit is now fully reactive — property changes update only the bound DOM nodes

## API

### `LitQuidElement`

Base class extending `LitElement`.

#### Methods

- `renderTemplate(): TemplateResult | typeof nothing` — Override to provide the component template. Used for CSR rendering and as the client-side template for post-hydration updates.

## Re-exports

The package re-exports the following `lit` primitives for convenience:

```typescript
export { html, svg, css, nothing, LitElement } from 'lit';
export type { TemplateResult, CSSResultGroup, PropertyValues, PropertyDeclaration } from 'lit';
export { customElement, property, state, query, /* ... */ } from 'lit/decorators.js';
```
